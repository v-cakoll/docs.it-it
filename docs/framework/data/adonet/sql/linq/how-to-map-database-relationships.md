# <a name="how-to-map-database-relationships"></a><span data-ttu-id="68509-102">Procedura: Eseguire il mapping delle relazioni di database</span><span class="sxs-lookup"><span data-stu-id="68509-102">How to: Map Database Relationships</span></span>
<span data-ttu-id="68509-103">Qualsiasi relazione tra i dati, che rimane prevedibilmente sempre la stessa, può essere codificata come riferimenti alla proprietà nella classe di entità.</span><span class="sxs-lookup"><span data-stu-id="68509-103">You can encode as property references in your entity class any data relationships that will always be the same.</span></span> <span data-ttu-id="68509-104">Nel database di esempio Northwind, ad esempio, poiché in genere i clienti effettuano ordini, nel modello è sempre presente una relazione tra clienti e ordini.</span><span class="sxs-lookup"><span data-stu-id="68509-104">In the Northwind sample database, for example, because customers typically place orders, there is always a relationship in the model between customers and their orders.</span></span>

> [!NOTE]
>  <span data-ttu-id="68509-108">I valori delle proprietà di archiviazione AssociationAttribute e ColumnAttribute rispettano la distinzione tra maiuscole e minuscole.</span><span class="sxs-lookup"><span data-stu-id="68509-108">AssociationAttribute and ColumnAttribute Storage property values are case sensitive.</span></span> <span data-ttu-id="68509-109">Verificare, ad esempio, che per i valori dell'attributo della proprietà AssociationAttribute.Storage venga usata la stessa combinazione di maiuscole e minuscole adoperata per i nomi di proprietà corrispondenti usati in altri punti del codice.</span><span class="sxs-lookup"><span data-stu-id="68509-109">For example, ensure that values used in the attribute for the AssociationAttribute.Storage property match the case for the corresponding property names used elsewhere in the code.</span></span> <span data-ttu-id="68509-110">Questo vale per tutti i linguaggi di programmazione .NET, anche quelli che non sono in genere tra maiuscole e minuscole, tra cui Visual Basic.</span><span class="sxs-lookup"><span data-stu-id="68509-110">This applies to all .NET programming languages, even those which are not typically case sensitive, including Visual Basic.</span></span> <span data-ttu-id="68509-111">Per altre informazioni sulla proprietà di archiviazione, vedere <xref:System.Data.Linq.Mapping.DataAttribute.Storage%2A?displayProperty=nameWithType>.</span><span class="sxs-lookup"><span data-stu-id="68509-111">For more information about the Storage property, see <xref:System.Data.Linq.Mapping.DataAttribute.Storage%2A?displayProperty=nameWithType>.</span></span>  
  
 <span data-ttu-id="68509-112">La maggior parte delle relazioni è di tipo uno-a-molti, come nell'esempio riportato più avanti in questo argomento.</span><span class="sxs-lookup"><span data-stu-id="68509-112">Most relationships are one-to-many, as in the example later in this topic.</span></span> <span data-ttu-id="68509-113">È anche possibile rappresentare relazioni uno-a-uno e molti-a-molti come segue:</span><span class="sxs-lookup"><span data-stu-id="68509-113">You can also represent one-to-one and many-to-many relationships as follows:</span></span>  
  
- <span data-ttu-id="68509-114">Uno a uno: Rappresentare questo tipo di relazione includendo <xref:System.Data.Linq.EntitySet%601> su entrambi i lati.</span><span class="sxs-lookup"><span data-stu-id="68509-114">One-to-one: Represent this kind of relationship by including <xref:System.Data.Linq.EntitySet%601> on both sides.</span></span>  
  
     <span data-ttu-id="68509-115">Si consideri, ad esempio, un `Customer` - `SecurityCode` relazione, creato in modo che il codice di sicurezza del cliente non verrà trovato nel `Customer` tabella ed è accessibile solo da utenti autorizzati.</span><span class="sxs-lookup"><span data-stu-id="68509-115">For example, consider a `Customer`-`SecurityCode` relationship, created so that the customer's security code will not be found in the `Customer` table and can be accessed only by authorized persons.</span></span>  
  
- <span data-ttu-id="68509-116">Molti-a-molti: In una relazione molti-a-molti, la chiave primaria della tabella dei collegamenti (anche denominato il *giunzione* tabella) è spesso formata da una combinazione di chiavi esterne delle altre due tabelle.</span><span class="sxs-lookup"><span data-stu-id="68509-116">Many-to-many: In many-to-many relationships, the primary key of the link table (also named the *junction* table) is often formed by a composite of the foreign keys from the other two tables.</span></span>  
  
     <span data-ttu-id="68509-117">Si consideri, ad esempio, un' `Employee` - `Project` relazione molti-a-molti formate usando tabella dei collegamenti `EmployeeProject`.</span><span class="sxs-lookup"><span data-stu-id="68509-117">For example, consider an `Employee`-`Project` many-to-many relationship formed by using link table `EmployeeProject`.</span></span> <span data-ttu-id="68509-118">In [!INCLUDE[vbtecdlinq](../../../../../../includes/vbtecdlinq-md.md)] è necessario che tale relazione venga modellata usando tre classi:`Employee`, `Project` e `EmployeeProject`.</span><span class="sxs-lookup"><span data-stu-id="68509-118">[!INCLUDE[vbtecdlinq](../../../../../../includes/vbtecdlinq-md.md)] requires that such a relationship be modeled by using three classes: `Employee`, `Project`, and `EmployeeProject`.</span></span> <span data-ttu-id="68509-119">In questo caso la modifica della relazione tra `Employee` e `Project` può apparentemente richiedere un aggiornamento della chiave primaria di `EmployeeProject`.</span><span class="sxs-lookup"><span data-stu-id="68509-119">In this case, changing the relationship between an `Employee` and a `Project` can appear to require an update of the primary key `EmployeeProject`.</span></span> <span data-ttu-id="68509-120">In questa situazione, tuttavia, è preferibile modellare la relazione eliminando una classe `EmployeeProject` esistente e creando una nuova classe `EmployeeProject`.</span><span class="sxs-lookup"><span data-stu-id="68509-120">However, this situation is best modeled as deleting an existing `EmployeeProject` and the creating a new `EmployeeProject`.</span></span>  
  
    > [!NOTE]
    >  <span data-ttu-id="68509-121">Le relazioni nei database relazionali vengono in genere modellate come valori di chiave esterna che fanno riferimento a chiavi primarie in altre tabelle.</span><span class="sxs-lookup"><span data-stu-id="68509-121">Relationships in relational databases are typically modeled as foreign key values that refer to primary keys in other tables.</span></span> <span data-ttu-id="68509-122">Per spostarsi tra di essi si associano in modo esplicito le due tabelle usando una relazionale *join* operazione.</span><span class="sxs-lookup"><span data-stu-id="68509-122">To navigate between them you explicitly associate the two tables by using a relational *join* operation.</span></span>  
    >   
    >  <span data-ttu-id="68509-123">Gli oggetti nello [!INCLUDE[vbtecdlinq](../../../../../../includes/vbtecdlinq-md.md)], da altra parte, fanno riferimento reciproco usando riferimenti alle proprietà o raccolte di riferimenti che è possibile spostarsi utilizzando *dot* notation.</span><span class="sxs-lookup"><span data-stu-id="68509-123">Objects in [!INCLUDE[vbtecdlinq](../../../../../../includes/vbtecdlinq-md.md)], on the other hand, refer to each other by using property references or collections of references that you navigate by using *dot* notation.</span></span>
