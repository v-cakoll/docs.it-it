# <a name="abstractions-abstract-types-and-interfaces"></a><span data-ttu-id="247f8-102">Astrazioni (interfacce e tipi astratti)</span><span class="sxs-lookup"><span data-stu-id="247f8-102">Abstractions (Abstract Types and Interfaces)</span></span>
<span data-ttu-id="247f8-103">Un'astrazione è un tipo che descrive un contratto, ma non fornisce un'implementazione completa del contratto.</span><span class="sxs-lookup"><span data-stu-id="247f8-103">An abstraction is a type that describes a contract but does not provide a full implementation of the contract.</span></span> <span data-ttu-id="247f8-104">Astrazioni sono in genere implementate come classi astratte o interfacce e diventano un set ben definito di documentazione di riferimento che descrive la semantica dei tipi che implementa il contratto richiesta.</span><span class="sxs-lookup"><span data-stu-id="247f8-104">Abstractions are usually implemented as abstract classes or interfaces, and they come with a well-defined set of reference documentation describing the required semantics of the types implementing the contract.</span></span> <span data-ttu-id="247f8-105">Alcune delle astrazioni più importanti in .NET Framework includono <xref:System.IO.Stream>, <xref:System.Collections.Generic.IEnumerable%601>, e <xref:System.Object>.</span><span class="sxs-lookup"><span data-stu-id="247f8-105">Some of the most important abstractions in the .NET Framework include <xref:System.IO.Stream>, <xref:System.Collections.Generic.IEnumerable%601>, and <xref:System.Object>.</span></span>  
  
 <span data-ttu-id="247f8-106">È possibile estendere Framework implementa un tipo concreto che supporta il contratto di un'astrazione e utilizzando questo tipo concreto con framework API consumer (che viene applicato) l'astrazione.</span><span class="sxs-lookup"><span data-stu-id="247f8-106">You can extend frameworks by implementing a concrete type that supports the contract of an abstraction and using this concrete type with framework APIs consuming (operating on) the abstraction.</span></span>  
  
 <span data-ttu-id="247f8-107">È molto difficile progettare un'astrazione significativa e utile che è in grado di resistere alla prova di tempo.</span><span class="sxs-lookup"><span data-stu-id="247f8-107">A meaningful and useful abstraction that is able to withstand the test of time is very difficult to design.</span></span> <span data-ttu-id="247f8-108">La difficoltà principale è ottenere il set corretto di membri, non è più e non un numero inferiore.</span><span class="sxs-lookup"><span data-stu-id="247f8-108">The main difficulty is getting the right set of members, no more and no fewer.</span></span> <span data-ttu-id="247f8-109">Se un'astrazione ha troppi membri, diventa difficile o persino impossibile da implementare.</span><span class="sxs-lookup"><span data-stu-id="247f8-109">If an abstraction has too many members, it becomes difficult or even impossible to implement.</span></span> <span data-ttu-id="247f8-110">Se è presente un numero troppo ridotto i membri per il funzionamento della promesso, diventa inutile in molti scenari interessanti.</span><span class="sxs-lookup"><span data-stu-id="247f8-110">If it has too few members for the promised functionality, it becomes useless in many interesting scenarios.</span></span>  
  
 <span data-ttu-id="247f8-111">Troppi astrazioni in un framework anche influire negativamente sull'usabilità del framework.</span><span class="sxs-lookup"><span data-stu-id="247f8-111">Too many abstractions in a framework also negatively affect usability of the framework.</span></span> <span data-ttu-id="247f8-112">È spesso piuttosto difficile da comprendere un'astrazione senza informazioni sulle modalità di integrazione in immagine più ampia delle implementazioni concrete e le API che viene applicato l'astrazione.</span><span class="sxs-lookup"><span data-stu-id="247f8-112">It is often quite difficult to understand an abstraction without understanding how it fits into the larger picture of the concrete implementations and the APIs operating on the abstraction.</span></span> <span data-ttu-id="247f8-113">Inoltre, i nomi di astrazioni e i relativi membri sono necessariamente astratti, che spesso li rende enigmatici e riducendone senza prima le informazioni sul contesto più ampio del loro utilizzo.</span><span class="sxs-lookup"><span data-stu-id="247f8-113">Also, names of abstractions and their members are necessarily abstract, which often makes them cryptic and unapproachable without first understanding the broader context of their usage.</span></span>  
  
 <span data-ttu-id="247f8-114">Tuttavia, astrazioni forniscono estensibilità estremamente potente che spesso non possono corrispondere a altri meccanismi di estendibilità.</span><span class="sxs-lookup"><span data-stu-id="247f8-114">However, abstractions provide extremely powerful extensibility that the other extensibility mechanisms cannot often match.</span></span> <span data-ttu-id="247f8-115">Sono alla base di molti schemi architetturali, ad esempio plug-in, inversione di controllo (IoC), le pipeline e così via.</span><span class="sxs-lookup"><span data-stu-id="247f8-115">They are at the core of many architectural patterns, such as plug-ins, inversion of control (IoC), pipelines, and so on.</span></span> <span data-ttu-id="247f8-116">Sono anche estremamente importante per la verificabilità del Framework.</span><span class="sxs-lookup"><span data-stu-id="247f8-116">They are also extremely important for testability of frameworks.</span></span> <span data-ttu-id="247f8-117">Astrazioni buona rendono possibile sottoporre a stub out pesante dipendenze allo scopo di unit test.</span><span class="sxs-lookup"><span data-stu-id="247f8-117">Good abstractions make it possible to stub out heavy dependencies for the purpose of unit testing.</span></span> <span data-ttu-id="247f8-118">In breve, le astrazioni sono tenute alle molteplici funzionalità più attese dei framework moderno e orientato.</span><span class="sxs-lookup"><span data-stu-id="247f8-118">In summary, abstractions are responsible for the sought-after richness of the modern object-oriented frameworks.</span></span>  
  
 <span data-ttu-id="247f8-119">**X DO NOT** forniscono astrazioni a meno che non vengono verificati per lo sviluppo di diverse implementazioni concrete e utilizzano le astrazioni di API.</span><span class="sxs-lookup"><span data-stu-id="247f8-119">**X DO NOT** provide abstractions unless they are tested by developing several concrete implementations and APIs consuming the abstractions.</span></span>  
  
 <span data-ttu-id="247f8-120">**✓ DO** scegliere con attenzione tra una classe astratta e un'interfaccia quando si progetta un'astrazione.</span><span class="sxs-lookup"><span data-stu-id="247f8-120">**✓ DO** choose carefully between an abstract class and an interface when designing an abstraction.</span></span>  
  
 <span data-ttu-id="247f8-121">**✓ CONSIDER** fornendo i test di riferimento per le implementazioni concrete di astrazioni.</span><span class="sxs-lookup"><span data-stu-id="247f8-121">**✓ CONSIDER** providing reference tests for concrete implementations of abstractions.</span></span> <span data-ttu-id="247f8-122">Questi test dovrebbero consentire agli utenti di verificare la corretta implementazione del contratto.</span><span class="sxs-lookup"><span data-stu-id="247f8-122">Such tests should allow users to test whether their implementations correctly implement the contract.</span></span>
