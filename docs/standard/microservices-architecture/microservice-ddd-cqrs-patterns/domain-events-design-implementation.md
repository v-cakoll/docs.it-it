## <a name="what-is-a-domain-event"></a><span data-ttu-id="ef516-108">Che cos'è un evento del dominio?</span><span class="sxs-lookup"><span data-stu-id="ef516-108">What is a domain event?</span></span>

<span data-ttu-id="ef516-109">Un evento è qualcosa che si è verificato in passato.</span><span class="sxs-lookup"><span data-stu-id="ef516-109">An event is something that has happened in the past.</span></span> <span data-ttu-id="ef516-110">Un evento del dominio è, logicamente, un evento che si è verificato in uno specifico dominio e che si vuole che altre parti dello stesso dominio (In-Process) riconoscano e a cui possano reagire.</span><span class="sxs-lookup"><span data-stu-id="ef516-110">A domain event is, logically, something that happened in a particular domain, and something you want other parts of the same domain (in-process) to be aware of and potentially react to.</span></span>

<span data-ttu-id="ef516-111">Un vantaggio importante di questo tipo di eventi è che gli effetti collaterali generati dopo un evento possono essere espressi in modo esplicito, anziché implicito.</span><span class="sxs-lookup"><span data-stu-id="ef516-111">An important benefit of domain events is that side effects after something happened in a domain can be expressed explicitly instead of implicitly.</span></span> <span data-ttu-id="ef516-112">Questi effetti collaterali devono essere coerenti in modo che avvengano tutte o nessuna delle operazioni relative all'attività di business.</span><span class="sxs-lookup"><span data-stu-id="ef516-112">Those side effects must be consistent so either all the operations related to the business task happen, or none of them.</span></span> <span data-ttu-id="ef516-113">Inoltre, gli eventi del dominio consentono una migliore separazione delle problematiche tra le classi all'interno dello stesso dominio.</span><span class="sxs-lookup"><span data-stu-id="ef516-113">In addition, domain events enable a better separation of concerns among classes within the same domain.</span></span>

<span data-ttu-id="ef516-114">Ad esempio, se si sta usa Entity Framework e le entità o persino le aggregazioni, in caso di effetti collaterali causati da un caso d'uso, questi verranno implementati come concetto implicito nel codice accoppiato dopo che si è verificato un evento.</span><span class="sxs-lookup"><span data-stu-id="ef516-114">For example, if you're just using Entity Framework and entities or even aggregates, if there have to be side effects provoked by a use case, those will be implemented as an implicit concept in the coupled code after something happened.</span></span> <span data-ttu-id="ef516-115">Tuttavia, se si esamina solo questo codice, si potrebbe non sapere se il codice (effetto collaterale) fa parte dell'operazione principale o è un effetto collaterale.</span><span class="sxs-lookup"><span data-stu-id="ef516-115">But, if you just see that code, you might not know if that code (the side effect) is part of the main operation or if it really is a side effect.</span></span> <span data-ttu-id="ef516-116">Al contrario, l'utilizzo di eventi del dominio rende il concetto esplicito e parte del linguaggio comune.</span><span class="sxs-lookup"><span data-stu-id="ef516-116">On the other hand, using domain events makes the concept explicit and part of the ubiquitous language.</span></span> <span data-ttu-id="ef516-117">Ad esempio, nell'applicazione eShopOnContainers la creazione di un ordine è un'operazione che non riguarda solo l'ordine. <span data-ttu-id="ef516-119">Gli eventi del dominio sono simili agli eventi di tipo messaggistica, con una differenza importante.</span><span class="sxs-lookup"><span data-stu-id="ef516-119">Domain events are somewhat similar to messaging-style events, with one important difference.</span></span> <span data-ttu-id="ef516-120">Con la messaggistica reale, l'accodamento di messaggi, i broker di messaggi o un bus di servizio che usa il protocollo AMPQ, un messaggio viene sempre inviato in modo asincrono e trasferito tra processi e computer.</span><span class="sxs-lookup"><span data-stu-id="ef516-120">With real messaging, message queuing, message brokers, or a service bus using AMPQ, a message is always sent asynchronously and communicated across processes and machines.</span></span> <span data-ttu-id="ef516-121">Questo approccio è utile per l'integrazione di più contesti delimitati, microservizi o persino applicazioni differenti.</span><span class="sxs-lookup"><span data-stu-id="ef516-121">This is useful for integrating multiple Bounded Contexts, microservices, or even different applications.</span></span> <span data-ttu-id="ef516-122">Tuttavia, con gli eventi del dominio si vuole generare un evento dall'operazione di dominio attualmente in esecuzione, facendo in modo che tutti gli effetti collaterali si verifichino all'interno dello stesso dominio.</span><span class="sxs-lookup"><span data-stu-id="ef516-122">However, with domain events, you want to raise an event from the domain operation you are currently running, but you want any side effects to occur within the same domain.</span></span>

<span data-ttu-id="ef516-123">Gli eventi del dominio e i relativi effetti collaterali, ossia le azioni attivate dopo l'elaborazione da parte dei gestori di eventi, devono verificarsi quasi immediatamente, in genere durante il processo, e all'interno dello stesso dominio.</span><span class="sxs-lookup"><span data-stu-id="ef516-123">The domain events and their side effects (the actions triggered afterwards that are managed by event handlers) should occur almost immediately, usually in-process, and within the same domain.</span></span> <span data-ttu-id="ef516-124">Gli eventi del dominio possono quindi essere sincroni o asincroni.</span><span class="sxs-lookup"><span data-stu-id="ef516-124">Thus, domain events could be synchronous or asynchronous.</span></span> <span data-ttu-id="ef516-125">Gli eventi di integrazione, invece, devono essere sempre asincroni.</span><span class="sxs-lookup"><span data-stu-id="ef516-125">Integration events, however, should always be asynchronous.</span></span>

## <a name="domain-events-versus-integration-events"></a><span data-ttu-id="ef516-126">Confronto tra eventi del dominio ed eventi di integrazione</span><span class="sxs-lookup"><span data-stu-id="ef516-126">Domain events versus integration events</span></span>

<span data-ttu-id="ef516-127">Da un punto di vista semantico gli eventi del dominio e quelli di integrazione si equivalgono, ovvero sono notifiche di un evento che si è appena verificato.</span><span class="sxs-lookup"><span data-stu-id="ef516-127">Semantically, domain and integration events are the same thing: notifications about something that just happened.</span></span> <span data-ttu-id="ef516-128">Tuttavia, la relativa implementazione deve essere diversa.</span><span class="sxs-lookup"><span data-stu-id="ef516-128">However, their implementation must be different.</span></span> <span data-ttu-id="ef516-129">Gli eventi del dominio non sono altro che messaggi inviati al dispatcher di eventi del dominio, eventualmente implementabile come mediatore in memoria in base a un contenitore IoC o a qualsiasi altro metodo.</span><span class="sxs-lookup"><span data-stu-id="ef516-129">Domain events are just messages pushed to a domain event dispatcher, which could be implemented as an in-memory mediator based on an IoC container or any other method.</span></span>

<span data-ttu-id="ef516-130">Al contrario, lo scopo degli eventi di integrazione è propagare le transazioni e gli aggiornamenti di cui è stato eseguito il commit in altri sottosistemi, indipendentemente se si tratti di altri microservizi, contesti delimitati o persino di applicazioni esterne.</span><span class="sxs-lookup"><span data-stu-id="ef516-130">On the other hand, the purpose of integration events is to propagate committed transactions and updates to additional subsystems, whether they are other microservices, Bounded Contexts or even external applications.</span></span> <span data-ttu-id="ef516-131">Tali eventi devono quindi verificarsi solo se l'entità è stata salvata in modo permanente poiché in molti scenari, in caso errore, l'intera operazione non si sarebbe mai verificata.</span><span class="sxs-lookup"><span data-stu-id="ef516-131">Hence, they should occur only if the entity is successfully persisted, since in many scenarios if this fails, the entire operation effectively never happened.</span></span>

<span data-ttu-id="ef516-132">Inoltre, come indicato, gli eventi di integrazione devono essere basati sulla comunicazione asincrona tra più microservizi (altri contesti delimitati) o persino applicazioni/sistemi esterni.</span><span class="sxs-lookup"><span data-stu-id="ef516-132">In addition, and as mentioned, integration events must be based on asynchronous communication between multiple microservices (other Bounded Contexts) or even external systems/applications.</span></span> <span data-ttu-id="ef516-133">Di conseguenza, l'interfaccia del bus di eventi richiede un'infrastruttura che consenta la comunicazione tra processi distribuita tra servizi potenzialmente remoti.</span><span class="sxs-lookup"><span data-stu-id="ef516-133">Thus, the event bus interface needs some infrastructure that allows inter-process and distributed communication between potentially remote services.</span></span> <span data-ttu-id="ef516-134">Può essere basata su un bus di servizio commerciale, sulle code, su un database condiviso usato come cassetta postale o qualsiasi altro sistema di messaggistica distribuito e preferibilmente di tipo push.</span><span class="sxs-lookup"><span data-stu-id="ef516-134">It can be based on a commercial service bus, queues, a shared database used as a mailbox, or any other distributed and ideally push based messaging system.</span></span>

    private readonly IIdentityService _identityService;

    public ValidateOrAddBuyerAggregateWhenOrderStartedDomainEventHandler(
        ILoggerFactory logger,
        IBuyerRepository<Buyer> buyerRepository,
        IIdentityService identityService)
    {
        // ...Parameter validations...
    }

    public async Task Handle(OrderStartedDomainEvent orderStartedEvent)
    {
        var cardTypeId = (orderStartedEvent.CardTypeId != 0) ? orderStartedEvent.CardTypeId : 1;        
        var userGuid = _identityService.GetUserIdentity();
        var buyer = await _buyerRepository.FindAsync(userGuid);
        bool buyerOriginallyExisted = (buyer == null) ? false : true;

        if (!buyerOriginallyExisted)
        {
            buyer = new Buyer(userGuid);
        }

        buyer.VerifyOrAddPaymentMethod(cardTypeId,
                                       $"Payment Method on {DateTime.UtcNow}",
                                       orderStartedEvent.CardNumber,
                                       orderStartedEvent.CardSecurityNumber,
                                       orderStartedEvent.CardHolderName,
                                       orderStartedEvent.CardExpiration,
                                       orderStartedEvent.Order.Id);

        var buyerUpdated = buyerOriginallyExisted ? _buyerRepository.Update(buyer) 
                                                                      : _buyerRepository.Add(buyer);

        await _buyerRepository.UnitOfWork
                .SaveEntitiesAsync();

        // Logging code using buyerUpdated info, etc.
    }
}
