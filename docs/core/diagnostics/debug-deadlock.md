# <a name="debug-a-deadlock-in-net-core"></a>Eseguire il debug di un deadlock in .NET Core

**Questo articolo si applica a: ✔️** .net core 3,1 SDK e versioni successive

In questa esercitazione si apprenderà come eseguire il debug di uno scenario di deadlock. Usando l'esempio fornito ASP.NET Core repository del codice sorgente dell' [app Web](https://docs.microsoft.com/samples/dotnet/samples/diagnostic-scenarios) , è possibile causare un deadlock intenzionalmente. L'endpoint riscontra un accumulo di blocchi e thread. Si apprenderà come usare diversi strumenti per analizzare il problema, ad esempio i dump di base, l'analisi dei dump di base e la traccia dei processi.

In questa esercitazione si apprenderà come:

> [!div class="checklist"]
>
> - Esaminare un blocco dell'app
> - Genera un file di dump di base
> - Analizzare i thread del processo nel file dump
> - Analizzare i blocchi di sincronizzazione e stack
> - Diagnosticare e risolvere un deadlock

## <a name="prerequisites"></a>Prerequisiti

L'esercitazione usa:

- [.NET Core 3,1 SDK](https://dotnet.microsoft.com/download/dotnet-core) o versione successiva
- [Esempio di destinazione di debug-app Web](https://docs.microsoft.com/samples/dotnet/samples/diagnostic-scenarios) per attivare lo scenario
- [DotNet-Trace](dotnet-trace.md) per elencare i processi
- [DotNet-dump](dotnet-dump.md) per la raccolta e l'analisi di un file dump

## <a name="core-dump-generation"></a>Generazione del dump di base

Per esaminare la mancata risposta dell'applicazione, un dump principale o un dump della memoria consente di controllare lo stato dei thread e di eventuali blocchi possibili che potrebbero presentare problemi di contesa. Eseguire l'applicazione di [debug di esempio](https://docs.microsoft.com/samples/dotnet/samples/diagnostic-scenarios) usando il comando seguente dalla directory radice dell'esempio:

```dotnetcli
dotnet run
```

Per trovare l'ID del processo, usare il comando seguente:

```dotnetcli
dotnet-trace ps
```

Prendere nota dell'ID del processo dall'output del comando. Il nostro ID processo era `4807` , ma il tuo sarà diverso. Passare all'URL seguente, che è un endpoint API nel sito di esempio:

[https://localhost:5001/api/diagscenario/deadlock](https://localhost:5001/api/diagscenario/deadlock)

La richiesta dell'API al sito si bloccherà e non risponde. Consentire l'esecuzione della richiesta per circa 10-15 secondi. Quindi creare il dump di base usando il comando seguente:

### <a name="linux"></a>[Linux](#tab/linux)

```bash
sudo dotnet-dump collect -p 4807
```

### <a name="windows"></a>[Windows](#tab/windows)

```console
dotnet-dump collect -p 4807
```

```dotnetcli
dotnet-dump analyze  ~/.dotnet/tools/core_20190513_143916
```

Stack sembra indicare che la richiesta è arrivata nel metodo deadlock che a sua volta effettua una chiamata a `Monitor.ReliableEnter` . Questo metodo indica che i thread tentano di accedere a un blocco di monitoraggio. Sono in attesa della disponibilità del blocco. È probabile che sia bloccato da un thread diverso.

Il passaggio successivo consiste nel scoprire quale thread sta effettivamente mantenendo il blocco del monitoraggio. Poiché i monitoraggi in genere archiviano le informazioni di blocco nella tabella dei blocchi di sincronizzazione, è possibile usare il `syncblk` comando per ottenere altre informazioni:

```console
> syncblk
Index         SyncBlock MonitorHeld Recursion Owning Thread Info          SyncBlock Owner
   43 00000246E51268B8          603         1 0000024B713F4E30 5634  28   00000249654b14c0 System.Object
   44 00000246E5126908            3         1 0000024B713F47E0 51d4  29   00000249654b14d8 System.Object
-----------------------------
Total           344
CCW             1
RCW             2
ComClassFactory 0
Free            0
```

Le due colonne interessanti sono **MonitorHeld** e le **informazioni sul thread proprietario**. La colonna **MonitorHeld** indica se un blocco di monitoraggio viene acquisito da un thread e dal numero di thread in attesa. Nella colonna **informazioni thread proprietario** viene visualizzato il thread attualmente proprietario del blocco di monitoraggio. Le informazioni sul thread hanno tre sottocolonne diverse. La seconda sottocolonna Mostra l'ID del thread del sistema operativo.

A questo punto, si conoscono due thread diversi (0x5634 e 0x51d4) che contengono un blocco di monitoraggio. Il passaggio successivo consiste nell'esaminare le operazioni svolte da tali thread. È necessario controllare se sono bloccati in modo indefinito. Usare i `setthread` `clrstack` comandi e per passare a ognuno dei thread e visualizzare il stack.

Per esaminare il primo thread, eseguire il `setthread` comando e individuare l'indice del thread 0x5634 (l'indice è 28). La funzione deadlock è in attesa di acquisire un blocco, ma è già proprietario del blocco. Si trova in un deadlock in attesa del blocco che possiede già.
